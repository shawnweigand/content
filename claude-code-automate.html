Section 04

Layer 01 — Model

4.1 — Haiku vs. Sonnet vs. Opus

Claude Haiku

Fast & Light

Fastest, cheapest, built for high-volume repetitive tasks. Use for routing, classification, parsing, and triage.

Best for → Repetitive tasks

Claude Sonnet ★

Balanced Workhorse

The default for most agentic work. Strong reasoning, solid speed, cost-effective for production pipelines.

Best for → Agentic workflows

Claude Opus

Deep Reasoning

Most capable. Complex reasoning, multi-constraint decisions, high-stakes outputs. Slower, pricier — worth it when it matters.

Best for → High-stakes outputs

4.2 + 4.3 — Extended Thinking & Effort Controls

🧠 Extended Thinking

Internal Reasoning Before Output

Claude works through the problem step-by-step internally before producing its response. Dramatically improves accuracy on hard problems, multi-constraint decisions, and deep logic.

⚙ /effort + Ultrathink

Control How Hard Claude Thinks

/effort sets the reasoning budget — low, medium, or high. Ultrathink pushes Claude to its maximum reasoning depth before responding. Use when the answer really matters.

Section 05

Layer 02 — Context

System Prompts

5.1 Prompts — System prompts define Claude's role before any user message. Set personas, enforce output formats, inject business rules, constrain scope. Makes Claude behave like a purpose-built internal tool, not a generic assistant.

Projects

5.2 Projects — Persistent home for a codebase or workflow. Holds conversation history, CLAUDE.md, and uploaded knowledge files. RAG retrieves relevant chunks so Claude always has the right context without re-pasting anything.

CLAUDE.md

5.3 — At project scope: encodes repo-specific conventions, architecture decisions, and task defaults. At user scope (~/.claude/CLAUDE.md): carries global preferences across all projects — stack, tone, code review style, shortcuts. Edit directly with # to update mid-session.

/memory

5.4 Auto Memory — Claude writes observations and decisions back to memory files mid-session. Use /memory to view and manage what Claude has stored. Builds a living record so every session starts smarter than the last.

/context

5.5 — Shows Claude exactly what is currently loaded in its context window — files, memory, conversation history, system prompt. Use it to understand what Claude knows right now and trim or add before a long task.

Section 06

Layer 03 — Tools

6.1 — Overview

What Tools Are

Tools give Claude hands — the ability to reach outside the conversation and act on the world. Without tools, Claude can only talk. With tools, it can do.

6.2 — MCP Servers

Model Context Protocol

Connects Claude to any external service as a first-class tool — Notion, Slack, GitHub, databases, internal APIs. Lightweight, composable, reusable. If it has an API, Claude can use it.

6.3 — CLI

Terminal Native

Claude runs terminal commands, reads output, reacts to errors, retries, patches code, reruns tests — in a loop, on your actual machine. Works the way a real engineer would.

Section 07

Layer 04 — Skills

7.1 — SKILL.md

Markdown Playbooks

Domain-specific instructions — deploy a service, write tests, generate a proposal, review a PR. Claude reads the file before acting. Consistent, auditable, version-controlled every time.

7.2 — Skill Creator

Build Skills with Claude

A meta-skill that instructs Claude how to design and write new skills. Describe a workflow, Claude produces the SKILL.md, refine it with evals. Skills are engineered over time, not just prompted once.

7.3 — Hooks

Event-Driven Triggers

Fire skills automatically at defined moments — before a task starts, after a file is written, when a test fails. Skills run in response to events, not just explicit commands.

Section 08

Layer 05 — Agents

8.1 — Subagents

Parallel Delegation

Claude instances spawned by a parent agent to handle specific subtasks in parallel. Orchestrator plans, delegates, synthesizes results. Complex jobs run concurrently — faster, fully autonomous.

8.2 — Agent Teams

Purpose-Built Roles

Purpose-built collections with defined roles — Planner, Coder, Reviewer, Deployer. Each carries its own context, tools, and skills. Complete workflows execute end-to-end with zero human handoff.

Section 09

Plugins

9.1 — Plugin Bundle

The Full Stack in a Box

A plugin is a bundle of everything — skills, hooks, agents, and MCP server configs packaged into a single installable unit. One command drops a complete, opinionated workflow into any project.

claude plugin add anthropics/official frontend-design
9.2 — Marketplace

Build Once, Share Everywhere

Publish plugins to a shared registry. Teams discover, install, and version-pin workflows. Community plugins cover popular stacks — Next.js, Notion, GitHub, Slack. Private registries keep internal workflows secure.

Drops the skill-creator SKILL.md and a set of eval templates into the project. Claude now knows how to build skills.

Step 11.2

Describe the Workflow

Tell Claude what the skill should do — in plain language. Claude uses the skill-creator SKILL.md to design the structure, write the instructions, define inputs and outputs, and produce a complete SKILL.md file.

Step 11.3

Refine and Ship

Claude runs evals against the new skill, identifies gaps, patches the instructions, and re-tests. The skill is engineered — versioned, tested, and ready to be added to a plugin or shared on the marketplace.
